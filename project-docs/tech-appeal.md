# 技術アピールポイント & 将来構想

---

## 現在の技術アピールポイント

---

### Redis によるセッション管理

会話の開始・中間・終了でアクセス先を使い分け、会話中のDB負荷をゼロに抑えている。

| タイミング | アクセス先 | 内容 |
|---|---|---|
| start | DB → Redis | 過去サマリ取得 → セッション保存 |
| turn | Redis のみ | 会話履歴の読み書き |
| end | Redis → DB | サマリ生成 → 永続化 → Redis 削除 |

---

### SSE によるリアルタイム二段階ストリーミング

1ターンに LLM を2回呼び、それぞれの応答をリアルタイムで順番にクライアントへ送信。

```
Phase 1：クローンの返答を生成 → 送信
Phase 2：野良アバターの返答を生成 → 送信
```

ユーザーは2キャラクターの発言が順番に流れてくるのをリアルタイムで見られる。

---

### ロール反転トリックで1つの LLM が両者を演じる

Gemini は会話履歴の role を `user` / `model` で管理する制約を持つ。  
クローン視点の返答を生成する際、**履歴のロールをまるごと反転**することで、追加の fine-tuning や別モデルなしに2キャラクターの対話を実現している。

```
通常の履歴：  model（野良）→ user（クローン）→ model（野良）
クローン生成：user（野良）→ model（クローン）→ user（野良）  ← 反転
```

---

### 会話でクローンの人格が成長する

会話終了時に LLM が会話ログを解析し、以下を構造化 JSON で出力する。

- **resonated**：相手が反応していたと感じた話題
- **newInterests**：初めて興味を示したキーワード
- **selfDiscovery**：この会話を通じた気づき
- **hobbyDiscoveries**：明らかに食いついたもの（厳しい確信度基準で判定）

`hobbyDiscoveries` に確信が持てた場合のみ clone の `likes` に追記される。  
**会話するたびにクローンの興味が実際に広がっていく。**

---

### 3層コンテキストビルディング

LLM に渡すプロンプトを毎回3層で構築し、過去の記憶を持った状態で会話を始める。

```
Layer 1：クローンプロフィール（MBTI・好き嫌い・自己紹介・探索タイプ）
Layer 2：直近7件のトピック履歴
Layer 3：過去エンカウントの記憶（印象・新関心・自己発見）
```

---

### Fallback パターン（デモ耐性）

LLM・DB のどちらが落ちても自動でモックエンジンに切り替わる。  
ネットワーク環境に依存せずデモが動く。

```
SupabaseEdgeFunctionImpl（本番）
    ↓ 失敗時
LLMMockImpl（モック）
```

---

### 3D バーチャルワールド（React Three Fiber）

- 自動巡回モード：ウェイポイント間を滑らかに補間移動
- 手動モード：タンク式操作（前後移動 + 左右旋回）
- エンカウント時：相手アバターが物理的に近づいてくるアニメーション
- カメラ：三人称追従 / 自由視点を状況に応じて切り替え
- Bloom + Vignette でサイバースペース的な雰囲気を演出

---

### クローンの好みでワールドが変わる

ユーザーの `likes` にマッチした部屋だけワールドに出現する。  
趣味が違えば見えるワールドも違う。世界そのものがパーソナライズされている。

---

## 技術的展望

---

### 現状の課題：コンテキストに中長期記憶がない

現在の `buildContext` は直近の情報しか持てない。

```
現状
├── クローンプロフィール（静的）
├── 直近7件のトピック（タイトルのみ）
└── 直近5件のエンカウントサマリ（それ以前は消える）
```

「この人はずっとこういう人だ」という中長期の記憶が欠落している。

---

### 将来構想①：階層記憶アーキテクチャ

記憶は人間と同様に、古くなるほど遠いストレージへ移行する階層を持たせる。

| 層 | ストレージ | 期間 | 内容 |
|---|---|---|---|
| Working Memory | Redis (TTL 30m) | 現セッション | 会話ターン履歴 |
| Episodic Memory | DB | 〜90日 | エンカウントサマリ・トピック |
| Semantic Memory | S3 + インデックス | 90日〜 | アーカイブ済み記憶 |
| Procedural Memory | DB（永続） | 永続 | likes / dislikes（会話で育つ人格） |

---

### 将来構想②：記憶のアーカイブフロー

90日以上アクセスのない記憶を S3 に転送し、DB にはインデックスのみ残す。  
「どんな記憶があるか」は検索できるまま、DB を軽量に保つ。

```
memory_index テーブル
├── topics[]        ← 検索用キーワード群
├── s3_key          ← null なら Warm、あれば Cold（S3）
├── summary_snippet ← ランキング用の短い抜粋
└── last_accessed_at
```

---

### 将来構想③：会話が記憶を呼び覚ます（非同期サイクル）

会話中の発言キーワードをもとに、アーカイブ済み記憶を非同期で検索・昇格させる。

```
turn 時（非同期）:
  発言キーワードを抽出
    → memory_index と照合
    → 関連する Cold 記憶を S3 から取得
    → Warm 層に昇格（last_accessed_at 更新）

次ターン以降:
  → 昇格した記憶がコンテキストに自動で入ってくる
```

**現在の話題と関連した過去の記憶が、会話を通じて浮かび上がってくる。**

---

### 将来構想④：レスポンス後の処理を非同期化

現状の `end` エンドポイントは、LLM 呼び出しや DB 更新をすべて同期で処理しており、ユーザーを数秒待たせている。

```
現状：LLM 呼び出し → DB INSERT → DB UPDATE → レスポンス（〜5秒）

改善：Redis 削除 → レスポンス（即時）
     after() で非同期 → LLM 呼び出し → DB INSERT → DB UPDATE
```

Next.js 15 の `after()` を使えば、レスポンス返却後にバックグラウンド処理を実行できる。  
将来の記憶アーカイブジョブも同じ仕組みに乗せられる。

---

### まとめ：現状 vs 将来

| 観点 | 現状 | 将来 |
|---|---|---|
| 短期記憶 | エンカウントサマリ直近5件 | Episodic Hot Layer |
| 中期記憶 | トピックタイトルのみ | Episodic Warm + reasoning |
| 長期記憶 | **なし** | Semantic Memory（S3 + インデックス） |
| 記憶の呼び起こし | **なし** | 非同期トピックマッチで昇格 |
| 人格の進化 | likes への書き込み | Procedural Memory として永続 |
| レスポンス速度 | 会話終了に数秒かかる | `after()` で即時返却 |
